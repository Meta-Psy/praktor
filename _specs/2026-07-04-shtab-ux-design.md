# Штаб: редизайн UI/UX Mission Control — design-doc

Дата: 2026-07-04. Статус: на ревью у Alex.
Проект: Meta-Psy/praktor (форк mtzanidakis/praktor), `ui/` + точечные изменения Go-бэкенда.

## 1. Цель и боли

Сделать веб-панель целостной, интуитивной и полноценно рабочей с телефона. Выявленные боли (по приоритету Alex):

1. **Хаос в навигации** — 14 разделов плоским списком без логики.
2. **Неудобные рабочие сценарии** — частые действия разбросаны; написать агенту из веба вообще нельзя.
3. **Визуальная сырость** — разнобой стилей: 3 разных паттерна подтверждений (один сломан — классы `.modal*` в Projects не определены), ~13 копий одинаковых инлайн-стилей, смесь русского и английского, отсутствие loading/empty-состояний на половине страниц, битые CSS-переменные (`--danger`, `--green-light`, `--bg-primary`…).

Основные сценарии (веб): контроль работы агентов; управление проектами/роадмапом; запуск работы из веба.

Выбранный подход — «Группировка + дизайн-система + чат» (средний путь): структура страниц сохраняется, радикальной перестройки в «рабочие пространства» нет.

## 2. Ключевые решения (утверждены Alex)

| Решение | Выбор |
|---|---|
| Язык интерфейса | Русский, везде. Технические термины (agent id, cron, MCP) не переводим насильно |
| Названия | «Штабная» метафора (см. §3) |
| Навигация | 3 группы по сценариям + слияние родственных страниц во вкладки (14 → 12 пунктов) |
| Визуальный стиль | «Индиго» — эволюция текущей тёмной темы с фиолетовым акцентом: те же цвета, единые радиусы/отступы/типографика. Светлая тема сохраняется |
| Мобильность | Наравне с десктопом, включая тач-поддержку граф-редактора |
| Новая функция | Чат с агентами из веба (единственное расширение объёма, требует backend) |

## 3. Навигация и названия

Сайдбар, заголовок панели — **«Штаб»** (вместо Mission Control):

```
Обстановка                  (Dashboard)
— Работа —
Связь                       (Conversations + отправка сообщений)
Отряды                      (Swarms)
Дежурства                   (Scheduled Tasks)
Приёмная                    (Intake + Plans, вкладки «Входящие» / «Планы»)
— Экосистема —
Операции                    (Projects)
Задачи                      (Portfolio)
Разведка                    (Radar + Intel, вкладки «Радар» / «Сводки»)
— Система —
Агенты                      (Agents: карточки, статусы, AGENT.md, расширения)
Арсенал                     (Catalog — каталог возможностей агентов)
Сейф                        (Secrets)
Досье                       (User Profile / USER.md)
```

- Слияния: `/intake` + `/plans` → одна страница «Приёмная» с вкладками; `/radar` + `/intel` → «Разведка» с вкладками. Старые маршруты редиректят на новую страницу с нужной вкладкой (закладки не ломаются).
- Информация о состоянии агентов живёт на «Агенты»; «Обстановка» её не дублирует.
- Группы в сайдбаре — заголовки-разделители, не сворачиваются (12 пунктов помещаются).

## 4. «Обстановка» (главная)

Из пассивных счётчиков — в центр решений. Утверждённый макет (`.superpowers/brainstorm/4568-1783110635/content/dashboard-v2.html`):

- **Шапка:** дата, версия, индикатор здоровья системы; строка компактных счётчиков (агенты в работе, активные дежурства, выполняющиеся отряды, новая почта).
- **«Требует решения»** (главная колонка) — карточки трёх типов, каждая со своими действиями по месту:
  - *План* (из Приёмной, ждёт подписи): проект, агент, когда поставлен, выдержка текста; действия «Подписать / Читать план / Отклонить».
  - *PR* (из Операций, готов к merge): репозиторий, статус CI, размер диффа; действия «Merge / Смотреть на GitHub». Сюда же — audit-issues с approve.
  - *Сбой* (упавшее дежурство): время, счётчик повторных сбоев, причина; действия «Повторить сейчас / перейти к источнику проблемы».
  - Клик по заголовку карточки ведёт на соответствующую страницу. Пусто → «Всё разобрано ✓».
- **«Активность»** (вторая колонка) — живая лента событий по WebSocket: ответы агентов, прогресс отрядов, деплои, merge, сбор разведсводок.

Данные: существующие `/api/status`, `/api/intake` (планы), `/api/projects` (PR/audit/deploy), `/api/tasks` (сбои), WS-события. Новых endpoint'ов не требуется: агрегация «требует решения» собирается на клиенте из перечисленных источников (они уже опрашиваются другими страницами).

## 5. «Связь» — чат с агентами из веба

**UX.** Страница Conversations становится полноценным чатом: слева агенты со статусом контейнера, справа переписка + поле ввода (Enter — отправить, Shift+Enter — перенос). Пока агент работает — индикатор «печатает…» и кнопка отмены (аналог `/stop`). Поиск по истории сохраняется. История общая с Telegram (та же таблица `messages`).

**Backend (Go).**
- `POST /api/agents/definitions/{id}/message` `{"text": "..."}` — кладёт сообщение в ту же очередь оркестратора, что и Telegram-сообщения: сохранение в `messages`, продолжение текущей сессии агента, автозапуск контейнера при необходимости.
- Ответ агента доставляется в UI по существующему WS (Conversations уже умеет инкрементально добавлять сообщения из событий `type=message`). **В Telegram ответ на веб-сообщение не дублируется** (прецедент — `RunCapture` из S6: синхронный ответ без TG-доставки).
- Отмена — существующий механизм `/stop` (control-топик), выведенный кнопкой.

**Вне объёма:** голос и файлы из веб-чата (только текст; голос/файлы остаются в Telegram).

## 6. Дизайн-система

**Библиотека** `ui/src/components/ui/`: `Button` (primary/secondary/danger), `Card`, `Badge`, `Input`/`Textarea`/`Select`, `Modal`, `ConfirmDialog`, `EmptyState`, `Spinner` + скелетоны, `Toast`, `Tabs`, `PageHeader`. Иконки — из App.tsx в отдельный модуль `ui/src/components/icons.tsx`.

**Стили:** дизайн-токены (цвет/радиус/отступ/шрифт) в CSS-переменных + CSS-файлы компонентов вместо инлайн-объектов. Определяется полный набор токенов — чинятся все битые ссылки (`--danger`, `--green-light`, `--bg-primary`, `--bg-secondary`, `--bg-main`, `--red-muted`). Стилевые константы из страниц (`card`, `btnPrimary`, `badge()` и т.д., продублированные в ~13 файлах) удаляются по мере миграции.

**Единые паттерны:**

| Паттерн | Правило |
|---|---|
| Подтверждение | Один `ConfirmDialog` для всех опасных действий: удаление отряда/секрета/дежурства, merge, approve, deploy, «очистить готовые». `window.confirm` и классовая модалка Projects удаляются. Добавляются отсутствующие подтверждения: удаление отряда, start/stop агента — не подтверждаем (безопасно), но toggle дежурства получает мгновенный откат при ошибке |
| Загрузка | Скелетон на первичной загрузке каждой страницы; `Suspense fallback` вместо `null` — общий спиннер |
| Пусто | `EmptyState` на каждой странице: что это за раздел, как начать пользоваться (закрывает обучение возможностям), с действием где уместно |
| Ошибки | `Toast` при ошибке действия; «Сохранено» — только после `res.ok` (чинится ложный «Saved» в Agents и Досье) |
| Связь | Индикатор WS — один, в сайдбаре (точка + подпись при обрыве); один экземпляр WS-подключения на приложение (контекст), страницы подписываются на него |
| Опрос | Интервалы поллинга унифицируются; где есть WS-события — поллинг убирается |
| Клавиатура | Кликабельные `div` → настоящие кнопки (tabIndex, Enter/Space); модалки: фокус-ловушка + Escape; toggle-бейджи → кнопки |

## 7. Мобильность и тач

- Полная адаптация всех страниц: сетки в 1 колонку, модалки bottom-sheet на весь экран, зоны нажатия ≥44px, `pre`-блоки со скроллом.
- **Граф-редактор отрядов:** mouse-события → pointer-события (мышь + палец едино); явный режим соединения (кнопка «Связать» → тап по двум узлам) вместо рисования линий перетаскиванием; пан/зум холста двумя пальцами; узлы перетаскиваются пальцем.
- «Связь» на телефоне — как мессенджер: список агентов сворачивается в выпадашку в шапке, поле ввода прижато к низу (с учётом клавиатуры).
- Off-canvas сайдбар и PWA-мета остаются как есть.

| Связь | Чат по §5 |

## 9. Тестирование и верификация

- Существующие vitest-тесты `*Status.ts` сохраняются и расширяются на новые хелперы (например, сборка списка «требует решения»).
- Компонентные тесты на ConfirmDialog/Toast/Tabs (vitest + testing-library) — минимальный набор на поведение, не на пиксели.
- Go: тесты на новый endpoint отправки сообщения (роутинг в очередь, запись в `messages`, отсутствие TG-доставки).
- Ручная верификация перед каждым PR: `npm run build` (ui), `CGO_ENABLED=0 go test ./internal/...`, прогон в браузере на десктопной и мобильной ширине.

## 10. Этапность (крупно — детали в implementation-плане)

1. **Фундамент:** токены + библиотека компонентов + иконки (ничего не ломает, страницы ещё старые).
2. **Каркас:** сайдбар «Штаба» (группы, русские названия, WS-индикатор), слияния страниц во вкладки, редиректы.
3. **Миграция страниц** на компоненты — пачками, по группам сайдбара (3–4 PR).
4. **Связь:** backend-endpoint + чат.
5. **Обстановка:** новая главная.
6. **Тач и мобильная полировка**, включая граф-редактор.

Каждый этап — отдельный PR в main через ревью Alex.

## 11. Вне объёма

- Голос/файлы в веб-чате.
- Перестройка в «рабочие пространства» (вариант B из обсуждения) — осознанно отложена.
- Синхронизация с upstream: редизайн диверсифицирует форк; конфликты с upstream принимаются как данность (форк живёт своей жизнью, upstream отслеживается через Разведку).
- Новые возможности бэкенда сверх §5.

## 12. Риски

- **Расхождение с upstream** — принято (см. §11).
- **Объём миграции 14 страниц** — режется этапами §10; страница мигрирует целиком за один PR, полусостояний нет.
- **Тач-редактор графа** — самая технически рискованная часть; если pointer-events + режим соединения окажутся недостаточными, допустимо упрощение мобильного варианта до просмотра топологии + текстового конструктора (как `@swarm` синтаксис в Telegram) — решение на этапе реализации с явным согласованием.
